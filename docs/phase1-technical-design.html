Personal Finance Manager

Phase 1 — Technical Design
Prepared for Harsh · June 4, 2026 · v0.1 · engineering blueprint for the Phase 1 PRD

This document translates the Phase 1 PRD into a buildable design: system architecture, data model, API surface, the data-ingestion approach, and — most importantly — how household visibility rules are enforced. It is implementation-leaning but stack-agnostic where the choice doesn't matter yet.

Phase 1 scope note: Phase 1 is a limited-user test release. Data is entered via document/statement upload and manual entry only — there is no live bank aggregation. Plaid aggregation is deferred to Phase 2 (§4 is retained as forward design).

Contents: 1. Architecture · 2. Data Model · 3. Visibility Enforcement · 4. Document Upload & Import (Phase 1) · 5. Plaid Integration (Phase 2) · 6. API Surface · 7. Security · 8. Future-Proofing · 9. Hosting Strategy · 10. Stack Notes

1. System Architecture

A conventional three-tier web app: a SPA client, a stateless API/service layer, a relational database, plus background workers for sync/enrichment and an integration boundary to Plaid.

  • Client: responsive SPA (the wireframed screens).
  • API/services: stateless; enforce authentication and the visibility/permission model on every request.
  • Database: PostgreSQL — relational fits the household/account/transaction structure and the integrity rules (safe-delete, dedup).

Notes on key entities

  • USER vs MEMBERSHIP. Login/identity lives on USER; role lives on MEMBERSHIP. Phase 1 enforces one active membership per user (see §8).
  • ACCOUNT. Scoped to a household_id and an owner_user_id. Visibility is an account attribute; only the owner can change it.
  • CATEGORY. Self-referential for sub-categories (parent_id); kind distinguishes income vs expense; is_system protects core categories (e.g. Income) from deletion.
  • SINKING_FUND. Linked to a category; holds the virtual reserve_balance; transactions that pay the bill reference it via sinking_fund_id (the B-6 marking).
  • TRANSACTION.dedup_hash. Deterministic hash (account + date + amount + normalized merchant) used to skip duplicates across imports (and Plaid sync in Phase 2).

3. Visibility Enforcement (the critical rule)

Visibility is enforced server-side at the data-access layer, never in the client. Every query for household data is scoped by the requesting user's membership and the per-account visibility rule.

Viewer contextSees account & its transactions?Counts in household totals?
Account ownerAlways (their own)Yes
Other member · account = SharedYesYes
Other member · account = PrivateNoNo
Other member · account = Balance-onlyBalance only (no line items)Yes (balance)
Implementation: a single authorization helper resolves, per request, the set of account IDs and the field-level rules the viewer may access; all repository queries take this scope as a mandatory parameter. Totals are computed from the same scope so a balance-only account contributes its balance without exposing transactions. This avoids per-endpoint ad-hoc checks (a common source of leaks).

4. Document Upload & Import Pipeline (Phase 1 — primary data path)

In Phase 1, all data enters via document/statement upload or manual entry. This pipeline is therefore a Phase 1 P0 capability.

  • Upload CSV/OFX/QFX to the encrypted object store. (PDF statements are a later enhancement — parsing is unreliable; not Phase 1.)
  • Detect account from header (masked number/institution); suggest target account; user confirms or creates a manual account.
  • Map columns → Date / Merchant / Amount (remembered per source for repeat imports).

Rule: if the account can't be matched with confidence, the user must pick before commit — no silent assignment.

4a. Transaction Categorization & Auto-Classification Engine

Implemented in packages/core/src/dedup.ts, apps/api/src/category/category.service.ts, and apps/api/src/transaction/transaction.service.ts. Runs after import (inline, Phase 1) and on-demand via the "Auto-classify" action on the Transactions page.

Merchant normalization pipeline

Every merchant string passes through two normalization steps before any matching occurs:

  1. normalizeMerchant(raw) — upper-case; strip legal suffixes (LLC, INC, CORP, LTD); collapse whitespace; remove all non-alphanumeric characters. Result: a clean, comparable token string.
  2. merchantRuleKey(raw) — runs normalizeMerchant first, then strips any trailing tokens that are purely numeric. Banks routinely append date codes, reference numbers, or store IDs to merchant names (e.g. "WF HOME MTG 06/09", "AMAZON MARKETPLACE 123"). After normalization these become "WF HOME MTG 0609" and "AMAZON MARKETPLACE 123"; stripping the trailing numeric token gives "WF HOME MTG" and "AMAZON MARKETPLACE" — stable keys that match across months and branches.
Rule: all rule storage and rule matching uses merchantRuleKey, not the raw or normalized string. This ensures a rule created from one month's import also fires for subsequent months' imports of the same merchant.

Multi-signal similarity scoring

merchantSimilarityScore(a, b): number — both inputs should already be processed by merchantRuleKey. Returns a value from 0 (no match) to 1 (exact). Three signals are computed; the maximum is returned so a strong match on any one signal is sufficient:

SignalFormulaCatches
Word Jaccard |A ∩ B| / |A ∪ B| Reordered words, extra tokens, synonymous suffixes
Containment fraction of the shorter string's words found in the longer One name is a prefix/abbreviation of the other; partial matches
Char-prefix ratio shared leading characters (spaces stripped) / max char length "WAL MART" ↔ "WALMART"; truncated names; OCR noise

Threshold: MERCHANT_MATCH_THRESHOLD = 0.65 — a pair scoring at or above this is considered the same merchant. Tunable; stored as an exported constant so a future UI setting or per-household override can adjust it.

Three-stage rule resolution

Used in applyRulesToAll (bulk classify) and applyRules (single transaction). Stages run in priority order; the first match above threshold wins.

  1. Explicit rules (CategoryRule table) — user-defined merchant → category mappings created via "Always categorize merchant as X" in the recategorize panel. The highest-scoring rule above threshold is used. Rules are stored as merchantRuleKey strings so a single rule covers all reference-code variants of that merchant.
  2. Learned from history (bulk classify only) — at classify time, all already-categorized transactions in the household are aggregated into a Map<merchantRuleKey, most-used categoryId>. The highest-scoring entry above threshold is used. Effect: one manual categorization of a merchant teaches the system for all future imports of that merchant — even without an explicit rule.
  3. Transfer pattern detection — a list of regexes (TRANSFER_PATTERNS in category.service.ts) matches common inter-account payment strings (ZELLE, VENMO, CREDIT CARD PAYMENT, AUTOPAY, ONLINE TRANSFER, etc.). Matched transactions are routed to the system Transfer category automatically, keeping them out of expense totals.

If no stage produces a match, the transaction remains uncategorized and surfaces in the "Needs review" tab for manual action.

Bulk classify performance

applyRulesToAll is designed to be O(transactions × rules) in memory, with only 4 parallel DB queries regardless of transaction count:

  1. All uncategorized transactions for visible accounts.
  2. All CategoryRule rows for the household.
  3. The system Transfer category.
  4. All already-categorized transactions (for the learned map).

Resolution runs entirely in memory; results are then bulk-written with one updateMany per distinct target category — not one update per transaction.

5. Plaid Integration (Phase 2 — forward design)

Deferred to Phase 2. Documented here so Phase 1 data structures (account source, dedup, categories) are built to accommodate it without rework. Plaid handles bank authentication; PFM never sees or stores bank credentials. The client uses Plaid Link; the backend exchanges and stores only the access token (encrypted).

sequenceDiagram
  participant U as User
  participant C as Client (Plaid Link)
  participant B as Backend
  participant P as Plaid
  U->>C: Tap "Connect"
  B->>P: create link_token
  P-->>B: link_token
  B-->>C: link_token
  C->>P: User authenticates (in Plaid UI)
  P-->>C: public_token
  C->>B: public_token
  B->>P: exchange public_token
  P-->>B: access_token (encrypt & store)
  B->>P: fetch accounts + transactions
  P-->>B: data
  B->>B: enrich · categorize · dedup · store
  Note over P,B: Webhooks notify of new txns / re-auth
  • Stored: Plaid access_token (encrypted) + item_id per connection; never bank credentials.
  • Sync: use Plaid's transactions sync; workers process deltas, enrich, categorize, and dedup.
  • Webhooks: handle SYNC_UPDATES_AVAILABLE and ITEM_LOGIN_REQUIRED (drives the reconnect/connection-health UI).
  • Categorization: map Plaid categories to household categories; reuse the same enrichment as import.

7. Security & Privacy

  • AuthN: email/password (hashed, e.g. argon2/bcrypt) + mandatory MFA (TOTP + email); short-lived sessions/tokens with refresh.
  • AuthZ: central permission helper; visibility scope applied to every data query (§3).
  • Encryption: TLS in transit; encryption at rest for DB and object store; secrets (Plaid tokens) encrypted with a managed KMS.
  • No bank credentials are ever stored (Plaid) — credential storage is explicitly out of the design.
  • Privacy: data export & deletion endpoints; consent tracked per connection; audit records for sensitive actions (member/role/visibility changes, exports).
  • Compliance: design toward SOC 2 readiness as scale grows. Engage qualified counsel/security specialists before handling real financial data.

9. Hosting Strategy

A deliberate two-stage approach: start cheap on Vercel for the test/beta phase, then migrate to a major cloud before launching with real user financial data at scale. The key is building portability seams now so the migration is infra + config, not an application rewrite.

9.3 Portability seams (build these in Stage 1)

  • Standard Postgres — migrates to Cloud SQL/RDS untouched (no Vercel/Neon-proprietary features).
  • S3-compatible storage (R2) behind a storage interface — swap to S3/GCS via config.
  • Job abstraction — background jobs behind a small interface so Inngest/QStash → Cloud Tasks/SQS is a swap, not a rewrite.
  • Framework-agnostic, containerizable backend — moving the API to Cloud Run/Fargate is straightforward.
  • IaC + GitHub Actions — reproducible environments; same CI/CD across both stages.
When to migrate: before opening to real user financial data at scale — that's when the stronger controls (managed KMS, SOC 2 alignment) and predictable cost at higher usage matter. Treat Vercel as the test/beta environment.

10. Stack Notes (recommendations, not mandates)

  • Database: PostgreSQL (relational integrity, mature, managed options everywhere).
  • Backend: any mainstream typed stack (e.g. Node/TypeScript, Python, Go) — pick by team familiarity; the design is framework-agnostic. Node/TS pairs naturally with Vercel in Stage 1.
  • Client: a modern SPA framework (Next.js fits the Vercel start); responsive-first.
  • Background work: a job queue + workers for imports & enrichment now; Plaid sync added in Phase 2.
Next deliverables from here: a build backlog (epics → stories) and, in parallel, monetization/pricing. The data model and API above are the contract the backlog builds against.
